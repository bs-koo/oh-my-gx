# oh-my-gx 프롬프트 예시집

실제 개발 상황별로 어떤 스킬을 사용하는지 안내합니다.
스킬 호출은 항상 `/gx-` 명령어를 사용하고, 옵션은 자연어로 표현할 수 있습니다.

---

## 1. 프로젝트 초기 세팅

### 처음 플러그인 설치 후

> **상황**: 프로젝트에 oh-my-gx를 처음 적용할 때

```
/gx-setup
```

**동작**: VCS 감지 → 필수 도구 확인 → 인증 설정을 단계별로 안내합니다.

### 코드베이스 스캔으로 도메인 자동 감지

> **상황**: 이미 코드가 있는 프로젝트에 플러그인을 처음 붙였을 때

```
/gx-context
```

**동작**: context/ 디렉토리가 없으면 코드베이스를 분석하여 도메인 구조를 자동 생성할지 묻습니다.

### 특정 도메인 직접 등록 (에너지-분석)

> **상황**: 에너지 분석 도메인을 Q&A로 등록하고 싶을 때

```
/gx-context 에너지-분석
```

**동작**: 5개 검증 질문(문제/프로세스/안 하면/규모/담당)을 통해 context를 생성합니다.

### 문서 기반 등록 (화면설계서, 요구사항 등)

> **상황**: PM이 정리한 에너지 모니터링 요구사항 문서가 있을 때

```
/gx-context 에너지-모니터링 문서 기반으로 등록해줘, 파일은 docs/energy-monitoring-req.md
```

**동작**: 문서를 읽고 내용을 분석하여 context를 생성합니다. PDF, 이미지도 지원합니다.

### 외부 규격 등록 (시큐어코딩 가이드 등)

> **상황**: 에너지 관련 법규 가이드나 시큐어코딩 가이드를 에이전트가 참조하게 하고 싶을 때

프로젝트 루트에 `references/` 디렉토리를 만들고 문서를 넣으면 됩니다.

```
mkdir references
cp ~/시큐어코딩가이드.pdf references/
cp ~/에너지이용합리화법-기술기준.pdf references/
```

**동작**: `/gx-dev` 실행 시 설계/구현/리뷰 에이전트가 자동으로 참조합니다.

## 2. 기능 개발 — 전체 파이프라인

### 새 기능 요청 (에너지 사용량 대시보드)

> **상황**: 건물별 에너지 사용량을 실시간으로 모니터링하는 대시보드를 처음부터 개발

```
/gx-dev 건물별 에너지 사용량을 실시간으로 조회할 수 있는 대시보드 추가해줘
```

**동작**: PRD 작성 → 사용자 승인 → 설계 → 사용자 승인 → 구현 → 자기점검 → 리뷰 → 커밋 → PR. 모드 선택 질문이 나오면 "전체 파이프라인"을 선택합니다.

### 이슈 키가 있는 요청

> **상황**: JIRA 티켓과 연결된 전력 수요 예측 기능 개발

```
/gx-dev [ENG-245] 전력 수요 예측 API 추가
```

**동작**: 브랜치명이 `ENG-245`로 자동 생성되고, 전체 파이프라인이 진행됩니다.

### PRD만 작성

> **상황**: 설비 가동률 분석 기능의 요구사항만 먼저 정리하고 싶을 때

```
/gx-dev 설비 가동률 분석 기능, PRD만 작성해줘
```

**동작**: PRD 작성 후 `.dev/{branch}/prd.md`에 저장하고 종료합니다.

### 설계만 작성

> **상황**: 탄소 배출량 추적 기능의 PRD는 있고 설계만 필요할 때

```
/gx-dev 설계만 해줘
```

**동작**: 기존 `.dev/{branch}/prd.md`를 읽어 설계를 진행합니다. PRD가 없으면 PRD부터 시작합니다.

## 3. 간단한 구현

### 설계 없이 바로 구현 (설정값 변경, 문구 수정 등)

> **상황**: 에너지 사용량 단위를 kWh에서 MWh로 변경하는 단순 작업

```
/gx-dev 대시보드의 에너지 사용량 표시 단위를 kWh에서 MWh로 변경해줘
```

**동작**: 모드 선택에서 "구현만"을 선택하면 설계/리뷰를 생략하고 바로 구현합니다.

처음부터 명시할 수도 있습니다:

```
/gx-dev 전력 피크 알림 임계값을 500kW에서 800kW로 변경, 구현만 해줘
```

### 긴급 버그 수정 (데이터 수집 오류 등)

> **상황**: 설비 센서 데이터 수집이 중단되는 버그를 빠르게 고쳐야 할 때

```
/gx-dev 설비 센서 데이터 수집 시 타임아웃으로 null 반환되는 버그 긴급 수정해줘
```

**동작**: "긴급", "핫픽스", "빨리 고쳐" 표현이 포함되면 hotfix 모드로 자동 전환됩니다. 경량 PRD → 구현 → 인수검증 → 커밋 → PR (설계/리뷰 생략).

## 4. 작업 이어하기

### 이전 작업 재개

> **상황**: 어제 하던 전력 수요 예측 작업을 이어서 하고 싶을 때

```
/gx-dev 이어서 해줘
```

또는:

```
/gx-dev 아까 하던 작업 계속해줘
```

**동작**: `.dev/*/state.md`에서 진행 중인 작업을 찾아 중단된 Phase부터 재개합니다.

### 여러 작업 중 선택

> **상황**: 진행 중인 작업이 여러 개일 때

```
/gx-dev 이어서 해줘
```

```
재개할 수 있는 작업이 여러 개 있습니다. 어떤 작업을 재개할까요?
→ feat/energy-dashboard — 에너지 사용량 대시보드
→ fix/sensor-timeout — 설비 센서 타임아웃 수정
```

**동작**: 사용자가 선택한 작업의 중단된 Phase부터 재개합니다.

### 진행 상태 확인

> **상황**: 현재 파이프라인이 어디까지 진행됐는지 확인

```
/gx-dev 지금 어디까지 됐어?
```

```
## 파이프라인 상태
- 작업: 전력 수요 예측 API
- 브랜치: ENG-245 (base: main)
- 현재 Phase: implement (in_progress)
```

**동작**: 현재 브랜치의 `.dev/{branch}/state.md`를 읽어 상태를 요약합니다.

### 다른 브랜치 기반

> **상황**: develop 브랜치에서 분기하여 작업

```
/gx-dev develop에서 탄소 배출량 월간 리포트 기능 추가해줘
```

**동작**: develop 브랜치를 base로 하여 새 작업 브랜치를 생성하고 파이프라인을 시작합니다.

## 5. 리뷰 / 커밋 / PR

### 리뷰만

> **상황**: 이미 구현한 에너지 사용량 집계 코드를 리뷰받고 싶을 때

```
/gx-dev 리뷰만 해줘
```

**동작**: qa-manager와 security-auditor가 병렬로 리뷰합니다.

### 커밋만

> **상황**: 변경사항을 커밋하고 싶을 때

```
/gx-commit
```

**동작**: 브랜치 타입을 파싱하고, diff를 분석하여 한국어 커밋 메시지를 자동 생성합니다.

### 커밋 메시지 직접 지정

> **상황**: 커밋 메시지를 직접 작성하고 싶을 때

```
/gx-commit 전력 피크 알림 임계값 설정 화면 추가
```

**동작**: 지정한 메시지로 커밋합니다. 브랜치 타입 접두어는 자동 부여됩니다.

### PR만

> **상황**: 커밋은 이미 했고 PR만 올리고 싶을 때

```
/gx-pull-request
```

**동작**: 커밋 히스토리를 분석하여 제목/본문을 자동 생성합니다.

### 커밋 + PR 체이닝

> **상황**: 커밋과 PR을 한 번에 처리하고 싶을 때

```
커밋하고 PR 올려줘
```

**동작**: `/gx-commit` 성공 후 자동으로 `/gx-pull-request`가 실행됩니다.

## 6. 현행 분석 + 영향도

### 정책 확인 (요약)

> **상황**: 현재 코드에서 에너지 요금 정산 정책이 어떻게 구현되어 있는지 확인

```
/gx-lens 에너지 요금 정산 정책 확인해줘
```

**동작**: 코드를 탐색하여 비즈니스 정책을 PO/PD가 이해할 수 있는 보고서로 제공합니다.

### 정책 확인 (상세)

> **상황**: 전력 피크 알림 로직을 더 깊이 파악하고 싶을 때

```
/gx-lens 전력 피크 알림 정책 확인해줘, 상세하게
```

**동작**: 더 많은 파일을 탐색하고 상세 보고서를 생성합니다.

### 정책 + 영향도 한 번에

> **상황**: 설비 가동률 기준을 변경하면 어디에 영향이 있는지 한 번에 확인

```
/gx-lens 설비 가동률 정책 확인해줘, 가동률 기준을 80%에서 70%로 낮추려고 해
```

**동작**: 보고서 출력 후 자동으로 영향도 분석(복잡도 + 리스크)을 수행합니다.

### 보고서만

> **상황**: 탄소 배출량 산정 정책을 정리만 하고, 영향도 분석은 나중에

```
/gx-lens 탄소 배출량 산정 정책 정리해줘
```

**동작**: 보고서 출력 후 "변경하려는 아이디어가 있으면 알려주세요"라고 묻습니다. "분석 없이 마치겠습니다"를 선택하면 종료합니다.

## 7. 도메인 관리

### 새 도메인 추가

> **상황**: 에너지 요금 정산 도메인을 새로 등록

```
/gx-context 에너지-정산
```

**동작**: 5개 검증 질문을 통해 도메인 context를 생성합니다.

### 기존 도메인 갱신

> **상황**: 에너지-분석 도메인에 새 정책이 추가되어 context를 갱신

```
/gx-context 에너지-분석
```

**동작**: `context/에너지-분석/`이 이미 존재하면 "어떤 내용을 갱신하시겠습니까?"를 묻습니다.

### 문서 기반 context 생성

> **상황**: PM이 정리한 설비 관리 화면설계서가 있을 때

```
/gx-context 파일에서 도메인을 추론해서 context 생성해줘, 파일은 requirements/설비관리-화면설계서.pdf
```

**동작**: 파일에서 도메인명을 추론하여 context를 생성합니다. 도메인명을 직접 지정할 수도 있습니다:

```
/gx-context 설비-관리 문서 기반으로 등록해줘, 파일은 docs/설비-요구사항.md
```

### 진행도 동기화

> **상황**: `/gx-dev`를 거치지 않고 개발한 내용을 status.md에 반영

```
/gx-context 에너지-분석 진행도 동기화해줘
```

**동작**: git 히스토리와 PR을 분석하여 status.md의 ⬜→✅ 갱신을 제안합니다.

## 8. 리서치

### 종합 리포트

> **상황**: 스마트그리드 기반 에너지 관리 시스템 트렌드를 조사

```
/gx-research 스마트그리드 기반 에너지 관리 시스템 트렌드
```

**동작**: 조사 깊이/결과물 형태를 묻습니다. "종합 리포트"와 "꼼꼼하게"를 선택하면 다수 소스를 교차 검증하여 리포트를 생성합니다.

### 비교 분석

> **상황**: 전력 수요 예측 모델을 비교하고 싶을 때

```
/gx-research LSTM vs Prophet 전력 수요 예측 모델 비교, 비교 분석 형태로
```

**동작**: 비교 기준 표 + 각 항목 요약 + 판단 근거를 포함한 비교 분석을 생성합니다.

### 핵심 요약

> **상황**: 국내 탄소 배출권 거래제 현황을 빠르게 파악

```
/gx-research 국내 탄소 배출권 거래제 현황, 요약만
```

**동작**: 한 줄 결론 + 핵심 포인트 3~5개로 요약합니다.

### 리서치 → context 반영

> **상황**: 리서치 결과를 도메인 context에 반영하고 싶을 때

리서치 완료 후 안내 메시지가 표시됩니다:

```
/gx-context 에너지-분석 문서 기반으로 갱신해줘, 파일은 .research/스마트그리드-기반-에너지-관리-시스템-트렌드-20260327.md
```

## 9. AI 글쓰기 교정

### 블로그 글 교정

> **상황**: 에너지 절약 관련 블로그 글에서 AI 티를 빼고 싶을 때

```
/gx-humanizer docs/energy-saving-blog.md
```

**동작**: 모드를 묻습니다. "rewrite"를 선택하면 AI 흔적을 제거하고 자연스러운 문체로 직접 수정합니다.

### 기술 문서 감사

> **상황**: 에너지 API 가이드 문서의 AI 흔적을 점검만 하고 싶을 때

```
/gx-humanizer docs/energy-api-guide.md
```

**동작**: "audit"를 선택하면 수정 없이 감지 리포트만 출력합니다:

```
## 감지 리포트
총 감지: 5건 (P1: 2건, P2: 2건, P3: 1건)

| # | 위치 | 심각도 | 패턴 | 원문 | 제안 |
|---|------|--------|------|------|------|
| 1 | L3   | P1     | K2   | "혁신적인 에너지 관리 방법론" | "이 방법" |
```

### 텍스트 직접 입력

> **상황**: 파일 없이 텍스트를 바로 교정하고 싶을 때

```
/gx-humanizer

오늘날 에너지 효율 향상의 중요성은 아무리 강조해도 지나치지 않습니다.
다양한 스마트그리드 기술의 발전으로 인해 혁신적인 변화가...
```

**동작**: 파일 대신 텍스트를 직접 붙여넣어도 됩니다. 동일하게 audit/rewrite 모드를 선택할 수 있습니다.

## 10. SVN 프로젝트

### SVN 환경 워크플로우

> **상황**: SVN 기반 프로젝트에서 oh-my-gx를 사용할 때

```
/gx-setup          ← VCS를 "SVN"으로 선택
/gx-context 에너지-분석    ← 도메인 등록 (동일하게 동작)
/gx-dev 설비 이상 감지 알림 기능 추가   ← PRD → 설계 → 구현 → 리뷰까지 진행
```

리뷰 완료 후 커밋은 직접 수행:

```
svn commit -m "feat: 설비 이상 감지 알림 기능 추가"
```

### SVN에서 지원되지 않는 것

- `/gx-commit` — `svn commit`을 직접 실행
- `/gx-pull-request` — SVN에는 PR 개념 없음
- 진행도 동기화 — PR 분석 불가, 커밋 로그만으로 동기화

## 11. 복합 시나리오

### context → 개발 → PR

> **상황**: 새 도메인을 등록하고, 기능을 개발하고, PR까지 한 번에 진행

```
① /gx-context 에너지-정산                              ← 도메인 지식 등록
② /gx-dev 에너지 요금 월간 정산 기능 추가해줘             ← 전체 파이프라인
   (PRD → 설계 → 구현 → 리뷰 → 커밋 → PR)
```

**동작**: 도메인 context가 등록된 상태에서 개발이 진행되므로, 에이전트가 용어와 아키텍처를 정확히 이해합니다.

### 리서치 → context → 개발

> **상황**: 전력 수요 예측 기술을 조사하고, 결과를 context에 반영한 뒤 기능을 개발

```
① /gx-research LSTM vs Prophet 전력 수요 예측 모델 비교, 비교 분석 형태로
② /gx-context 에너지-분석 문서 기반으로 갱신해줘, 파일은 .research/lstm-vs-prophet-전력-수요-예측-모델-비교-20260327.md
③ /gx-dev 전력 수요 예측 API 구현해줘
```

**동작**: 리서치 결과가 도메인 context에 반영되어, 개발 시 에이전트가 기술 선택 근거를 참조합니다.

### 현행 분석 → 영향도 → 개발

> **상황**: 설비 가동률 기준을 변경하기 전에 현황을 파악하고, 영향도 확인 후 개발

```
① /gx-lens 설비 가동률 정책 확인해줘
   → 영향도 질문에 "가동률 기준을 80%에서 70%로 낮추려고 해" 입력
② /gx-dev 설비 가동률 기준을 70%로 변경해줘
```

**동작**: 영향도 분석 결과를 참고하여, 어떤 모듈이 영향을 받는지 인지한 상태로 개발을 진행합니다.

### 리뷰 → 커밋 → PR

> **상황**: 기존 코드를 리뷰받고, 문제없으면 커밋과 PR까지 이어서 진행

```
① /gx-dev 리뷰만 해줘
② 커밋하고 PR 올려줘
```

**동작**: 리뷰 완료 후 `/gx-commit`이 실행되고, 성공하면 자동으로 `/gx-pull-request`가 이어집니다.

## 12. 같은 요청, 다양한 표현

동일한 기능을 요청하더라도 표현 방식은 자유롭습니다. 아래는 모두 같은 동작을 수행합니다.

```

```
/gx-dev 전력 사용량이 임계값을 넘으면 알림 발송하는 기능 만들어줘

```

```

```

```

```

```
/gx-dev 데이터 수집 장애 핫픽스
```

### 현행 분석 요청

```

```

```

```

```
