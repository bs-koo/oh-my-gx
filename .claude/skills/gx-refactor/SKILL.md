# gx-refactor

> **이 스킬**: refactor — REFACTOR 단계 (안전한 정리)
> **호출 시 주의**: 이 스킬 내에서 다른 스킬을 호출할 때 반드시 `oh-my-gx:` 접두사를 사용한다.

GREEN 상태(모든 테스트 통과)를 유지하며 중복 제거, 가독성 향상 등 정리를 수행한다. **동작 변경 절대 금지.**

---

## Iron Law

```
NO BEHAVIOR CHANGES DURING REFACTOR
```

위반 시:
- 변경 사항 즉시 롤백
- 동작 변경이 필요하면 새 RED 단계로 진입
- "조금 더 좋게 동작하도록" 금지

---

## 사용 시점

- 보조 스킬 체이닝: `oh-my-gx:gx-green` 완료 직후 자동 호출 (gx-tdd 파이프라인은 이 스킬 대신 refactor-coder 에이전트를 직접 디스패치)
- GREEN 선행 없이 단독 호출 금지 (GREEN 상태가 이미 있으면 단독 호출 가능 — skill-routing의 'REFACTOR 단계' 단독 호출은 이 경우를 가리킨다)
- 호출 시 GREEN 상태 검증:
  ```
  대상 테스트가 통과하는가? Bash로 테스트 실행하여 확인.
  실패면: "REFACTOR는 GREEN 상태가 필요합니다. oh-my-gx:gx-green 먼저 호출하세요." 중단.
  ```

## 실행 절차

### Step 0: 정체성 확인

```
oh-my-gx:gx-refactor — REFACTOR 단계 진입.
GREEN 상태 유지하며 정리만 수행합니다.
동작 변경 절대 금지. 테스트 깨지면 즉시 롤백.
```

### Step 1: GREEN 상태 검증

```bash
./gradlew test  # 또는 npm test
```

- ✅ 모든 테스트 통과 → 정상. Step 2 진행
- ❌ 일부 실패 → "GREEN 상태가 아닙니다. green 단계로 복귀하세요." 중단

### Step 2: 정리 대상 식별

GREEN 단계에서 작성된 코드를 분석하여 정리 대상을 식별한다.

정리 대상:
- **중복**: 동일 로직이 2회 이상 반복
- **명명**: 의도가 불분명한 변수/함수명
- **구조**: 한 함수가 너무 많은 일을 함
- **추출**: 인라인 로직을 헬퍼로 분리

정리 비대상 (REFACTOR에서 하지 않음):
- 동작 변경 (값 변경, 로직 변경)
- 새 기능 추가
- 에러 핸들링 추가
- 성능 최적화 (별도 작업)

### Step 3: 정리 수행 (refactor-coder 디스패치)

격리된 서브에이전트 호출:

```
Task(subagent_type="oh-my-gx:refactor-coder"):
  description: "Refactor: {정리 대상}"
  prompt: |
    당신은 REFACTOR 단계 정리 전담자입니다.

    [중요] 다음 규칙을 절대 준수합니다:
    1. 동작을 변경하지 않습니다. 입력→출력 매핑이 동일해야 합니다.
    2. 새 기능을 추가하지 않습니다.
    3. 매 정리 후 테스트를 실행하여 GREEN 상태를 유지합니다.
    4. 테스트가 깨지면 즉시 변경을 되돌립니다.

    정리 대상:
    {대상 파일과 정리 항목}

    수행 가능한 정리:
    - 중복 제거 (Extract Method)
    - 변수/함수 이름 개선 (Rename)
    - 구조 정리 (Extract Class, Move Method)
    - 매직 넘버 상수화
    - 테스트 코드 정리 (모의 동작 검증을 실제 동작 검증으로 교체, 테스트 전용 프로덕션 메서드를 테스트 유틸리티로 이동 — 검증 강도를 낮추지 않는 범위)

    수행 절차:
    1. 정리 1건 수행
    2. 테스트 실행
    3. 통과 확인 → 정리 1건 더 수행
    4. 실패 확인 → 즉시 롤백 → 다른 정리 시도

    작업이 끝나면 다음을 보고:
    - 정리 항목 목록 (각 항목당 테스트 통과 확인)
    - 변경된 파일 경로
    - 최종 테스트 결과
```

### Step 4: 최종 검증

refactor-coder 보고 후 직접 검증:

1. 전체 테스트 실행
2. **모든 테스트 통과** 확인
3. 동작 변경 감지:
   - public 인터페이스 시그니처 변경 → 동작 변경 가능성 → 사용자 확인
   - 새 메서드 추가 → 호출자가 늘었는지 확인 (정리는 호출자 변경 없음이 원칙)

**Iron Law 위반 감지**:
- 테스트 실패 → 즉시 롤백 요구. refactor-coder 재호출 또는 사용자에게 보고
- 새 동작 추가 감지 → "REFACTOR 범위 위반. 새 동작은 새 RED 단계로 진입하세요." 안내

### Step 5: 사이클 완료

```
REFACTOR 단계 완료:
- 정리: {N건}
- 테스트: 전체 통과
- 동작 변경: 없음

이번 TDD 사이클 완료. 다음 컴포넌트로 진행.
```

## 합리화 격파 표

| 변명 | 반박 |
|------|------|
| "조금만 더 좋게 동작하도록" | 동작 변경은 새 RED 단계. REFACTOR는 정리만 |
| "이왕 손댄 김에 버그 하나 수정" | 버그도 새 RED 단계. 별도 사이클로 |
| "리팩토링 하다 보니 새 기능이 보임" | 메모만 하고 정리 완료 후 새 RED로 |
| "테스트가 좀 어색하지만 동작은 OK" | 테스트 어색함도 REFACTOR 대상. 단, 동작은 동일 유지 |
| "이 정도는 동작이 같은 거나 마찬가지" | "마찬가지"가 아니다. 테스트가 깨지면 같지 않다 |

---

## Red Flags

- "이 부분 동작을 조금만 바꾸면 더 좋을 듯"
- "버그 같은 거 발견 — 슬쩍 수정"
- "테스트가 너무 엄격해서 약화"
- "주석이 헷갈리니 코드도 같이"

→ 모두 Iron Law 위반 신호. 정리 범위 밖. 새 RED 단계로 진입.

## 다른 스킬 호출 시 절대 규칙 (Iron Law)

✅ 올바름:
- `Skill("oh-my-gx:gx-red")` — 새 사이클 진입 (필요 시)
- `Skill("oh-my-gx:gx-verify")` — 사이클 완료 후 검증

❌ 금지:
- `Skill("red")`, `Skill("verify")` — 접두사 누락

**위반 시**: 즉시 중단하고 fully-qualified 이름으로 재호출.
