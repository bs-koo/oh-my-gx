# TDD Iron Law 카탈로그

gx-tdd 파이프라인의 모든 단계는 아래 Iron Law를 따른다. **문자 위반 = 정신 위반.**

---

## 3대 Iron Law

```
1. NO PRODUCTION CODE WITHOUT A FAILING TEST FIRST
   (실패 테스트 없이 프로덕션 코드 작성 금지)

2. NO PHASE SKIPPING WITHOUT --hotfix
   (--hotfix 모드 외에 Phase 건너뛰기 금지)

3. NO COMPLETION CLAIMS WITHOUT FRESH VERIFICATION EVIDENCE
   (신선한 검증 증거 없이 완료 주장 금지)
```

위반 시:
- Iron Law 1 위반 → 코드 삭제 후 RED 단계부터 재시작
- Iron Law 2 위반 → 건너뛴 Phase부터 재실행
- Iron Law 3 위반 → verify 게이트로 복귀

**예외 없음.** "이번 한 번만"은 첫 예외가 규칙이 된다.

---

## 합리화 격파 표

TDD 강제 시 발견되는 23가지 변명과 반박. 자가 점검 시 또는 사용자에게 안내 시 사용한다.

### 그룹 1: 단순함 합리화

| 변명 | 반박 |
|------|------|
| "이건 너무 간단해서 테스트 필요 없음" | 단순한 코드도 깨진다. 테스트 30초면 끝 |
| "한 줄짜리 함수에 테스트는 과함" | 한 줄짜리에서 가장 많은 버그가 나온다 |
| "getter/setter는 테스트 불필요" | getter/setter는 보통 안전하지만, AC에 포함되면 테스트한다 |

### 그룹 2: 사후 테스트 합리화

| 변명 | 반박 |
|------|------|
| "이미 수동으로 테스트했음" | 수동은 재실행 불가. 자동 테스트만 신뢰 가능 |
| "테스트 나중에 작성해도 같은 효과" | 사후 = "이게 뭐 하는 코드?", 사전 = "이게 뭘 해야 함?" |
| "Tests after achieve the same goals" | 사후 테스트는 구현에 편향됨. 사전 테스트는 요구사항 발견 |
| "30분 테스트 작성이 TDD와 같음" | 커버리지는 같아도, 사전이 깨질 가능성을 미리 본다 |
| "테스트를 고치면 금방 통과하는데" | 테스트 수정은 GREEN의 실패 모드. 코드를 고쳐라 |

### 그룹 3: 매몰비용 합리화

| 변명 | 반박 |
|------|------|
| "X시간 작성한 코드 버리는 건 낭비" | 매몰비용 오류. 검증 안 된 코드 유지가 더 큰 부채 |
| "참조용으로만 코드 보관" | 보면 적응한다. 그건 사후 테스트. 삭제는 삭제 |
| "비슷한 코드를 다시 작성하는 건 비효율" | 두 번째 작성은 첫 번째보다 항상 빠르고 좋다 |

### 그룹 4: 실용주의 합리화

| 변명 | 반박 |
|------|------|
| "TDD는 교조적, 실용주의자는 유연하게" | TDD가 실용적이다. 디버깅 시간이 더 빠르다 |
| "정신을 따르면 됨, 문자는 무시" | 문자를 무시하면 정신도 무시된다 |
| "이번 한 번만 예외" | 첫 예외가 규칙이 된다 |
| "프로젝트 일정이 급함" | 사후 디버깅이 더 오래 걸린다 |

### 그룹 5: 확신 합리화

| 변명 | 반박 |
|------|------|
| "이 케이스는 명백히 동작함" | "should work"는 검증이 아니다. 실행 증거 필요 |
| "리뷰어가 어차피 잡아낼 것" | 리뷰어 부담 떠넘기기. 본인이 먼저 검증 |
| "다른 곳에 비슷한 테스트가 있음" | "비슷"은 검증이 아니다. 이 케이스만의 테스트 필요 |

### 그룹 6: 환경 합리화

| 변명 | 반박 |
|------|------|
| "기존 코드에 테스트가 없음" | 새 코드에는 테스트를 추가한다 |
| "테스트 환경이 복잡함" | 환경 셋업도 TDD의 일부. 한 번 만들면 재사용 |
| "테스트 작성이 코드 작성보다 어려움" | 테스트가 어려우면 설계가 어렵다. 인터페이스를 단순화하라 |
| "프레임워크 제약으로 테스트 불가" | 거의 항상 우회 가능. 가능 검증 전 포기 금지 |

### 그룹 7: 시간 합리화

| 변명 | 반박 |
|------|------|
| "TDD는 시간이 두 배" | 디버깅 시간이 절반 이하. 총 시간 단축 |
| "지금은 빠르게 끝내고 나중에 리팩토링" | 사후 리팩토링은 거의 안 일어난다 |

## Red Flags 자가 점검

다음 생각이 들면 합리화 중이다. **STOP 신호.**

| 생각 | 의미 | 조치 |
|------|------|------|
| "테스트 먼저 작성하면 시간이 두 배" | 시간 합리화 | 시간 합리화 반박 표 참조 |
| "이 케이스는 명백히 동작함" | 확신 합리화 | 실행 증거 요구 |
| "이미 비슷한 테스트가 있음" | 확신 합리화 | 이 케이스만의 테스트 작성 |
| "PR 일정이 급함" | 실용주의 합리화 | --hotfix 모드 사용 (전체 우회 아님) |
| "리뷰어가 어차피 잡아낼 것" | 확신 합리화 | 본인이 먼저 verify |
| "테스트 환경이 복잡함" | 환경 합리화 | 환경 셋업도 작업의 일부 |
| "이번 한 번만" | 실용주의 합리화 | 첫 예외가 규칙이 된다 |
| "코드는 명백히 옳음" | 확신 합리화 | "should" 사용 = 검증 부족 |
| "테스트보다 더 빠르게 수동 검증 가능" | 사후 테스트 합리화 | 수동은 재실행 불가 |

→ 모두 STOP 신호. RED 단계부터 다시 시작 또는 verify 게이트 호출.

## Iron Law 위반 감지 패턴

스킬 실행 중 다음 패턴을 감지하면 즉시 중단한다.

### "should/probably" 어휘 사용

```
❌ "테스트가 통과할 것 같음"
❌ "이 변경으로 버그가 고쳐졌을 것 같음"
❌ "should work"
❌ "looks good to me"
```

→ Iron Law 3 위반. verify 게이트 호출.

### 만족 표현

```
❌ "완료!"
❌ "Great!"
❌ "Perfect!"
❌ "Done!"
```

→ 검증 증거 없으면 Iron Law 3 위반.

### Phase 스킵 시도

```
❌ "요구사항이 명확하니 PRD 생략"
❌ "간단하니까 설계 생략"
❌ "이미 비슷한 거 만들어봤으니 RED 생략"
```

→ Iron Law 2 위반. 건너뛴 Phase부터 재실행.

### 코드 우선 작성

```
❌ "일단 구현하고 테스트는 나중에"
❌ "이 함수는 단순하니까 테스트 없이 시작"
❌ "프로토타입이니까 TDD 생략"
```

→ Iron Law 1 위반. 코드 삭제 후 RED 단계부터 재시작.

## 적용 우선순위

여러 위반이 동시 발생 시 처리 순서:
1. Iron Law 1 (코드 우선 작성) — 가장 심각. 코드 삭제 후 재시작
2. Iron Law 3 (검증 없는 완료 주장) — verify 게이트 호출
3. Iron Law 2 (Phase 스킵) — 건너뛴 Phase부터 재실행

---

## 사용자 안내 톤

Iron Law 위반을 사용자에게 알릴 때 다음 톤을 따른다:

✅ 권장: "Iron Law 1 위반을 감지했습니다. 코드를 작성하기 전에 실패 테스트를 먼저 작성해야 합니다. RED 단계부터 다시 시작합니다. (이유: {합리화 격파 표 인용})"

❌ 금지: "당신이 잘못했습니다", "왜 그러셨나요?"

위반은 사용자가 의도한 게 아니라 합리화가 자동으로 작동한 결과다. 책임 추궁이 아니라 패턴 인식과 즉시 교정이 목적이다.
