# gx-red

> **이 스킬**: red — RED 단계 (실패 테스트 우선 작성)
> **호출 시 주의**: 이 스킬 내에서 다른 스킬을 호출할 때 반드시 `oh-my-gx:` 접두사를 사용한다.

실패 테스트를 우선 작성하고 실패 확인까지 책임진다. **Iron Law**: 실패 테스트 없이 어떤 프로덕션 코드도 작성되지 않는다.

---

## Iron Law

```
NO PRODUCTION CODE WITHOUT A FAILING TEST FIRST
```

위반 시:
- 작성된 프로덕션 코드 즉시 삭제
- RED 단계부터 재시작
- "참조용으로 보관" 금지. 삭제는 삭제

---

## 사용 시점

- `oh-my-gx:gx-tdd` 파이프라인의 red-green-refactor Phase 내부 자동 호출
- 단독 호출: 사용자가 "RED 단계" 또는 "실패 테스트 먼저"를 명시

---

## 실행 절차

### Step 0: 정체성 확인

오케스트레이터에게 알린다:

```
oh-my-gx:gx-red — RED 단계 진입.
이 시점부터 어떤 프로덕션 코드도 작성하지 않습니다.
실패 테스트 작성 → 실패 확인 → green 단계로 인계.
```

### Step 1: 테스트 대상 확인

다음 정보를 확보한다:
- **AC (수용 기준)**: `${DEV_DIR}/prd.md`의 Given-When-Then 시나리오
- **설계서**: `${DEV_DIR}/design.md`의 testability 평가
- **대상 컴포넌트**: 어떤 클래스/함수의 어떤 동작을 테스트하는가
- **테스트 프레임워크**: 프로젝트의 기존 테스트 스타일

단독 호출 시 위 정보가 없으면 사용자에게 질문:
```
AskUserQuestion(
  question: "어떤 동작을 테스트할까요?",
  description: "예: 잘못된 비밀번호 로그인 시 401 응답"
)
```

### Step 2: 실패 테스트 작성 (red-writer 디스패치)

격리된 서브에이전트 호출:

```
Task(subagent_type="red-writer"):
  description: "Write failing test for: {AC}"
  prompt: |
    당신은 RED 단계 테스트 작성 전담자입니다.

    [중요] 다음 규칙을 절대 준수합니다:
    1. 프로덕션 코드를 작성하지 않습니다. 테스트만 작성합니다.
    2. 기존 프로덕션 코드를 보지 않습니다. AC와 설계서만 봅니다.
    3. 테스트가 반드시 실패해야 합니다 (대상 함수가 아직 없거나 빈 구현).

    AC: {Given-When-Then 시나리오}
    설계서 testability 섹션: {대상 컴포넌트의 인터페이스 + 모의 전략}
    기존 테스트 스타일: {프로젝트의 테스트 컨벤션}

    작업:
    1. AC를 검증하는 최소 테스트 1개 작성
    2. 테스트 명령으로 실패 확인 (에러 메시지 캡처)
    3. 테스트 파일 경로, 테스트 코드, 실패 메시지를 보고

    출력 형식:
    - 테스트 파일: {경로}
    - 테스트 코드: {코드}
    - 실패 확인 명령: {명령}
    - 실패 메시지: {메시지}
```

### Step 3: 실패 확인 (verification)

red-writer 보고 후 직접 검증:

1. 테스트 명령 실행 (`./gradlew test --tests <테스트>` 또는 동등 명령)
2. **실패** 확인 (통과하면 잘못된 테스트)
3. 실패 사유 확인:
   - ✅ 대상 함수가 없음 (NoSuchMethodError 등)
   - ✅ 빈 구현이라 assertion 실패
   - ❌ 컴파일 에러 (테스트 자체가 잘못됨) → red-writer 재호출
   - ❌ 다른 테스트도 실패 (영향 범위 다름) → 격리 확인

**Iron Law 위반 감지**:
- 테스트가 즉시 통과하면 → 이미 구현이 있다는 뜻. 사용자에게 보고하고 중단:
  ```
  "테스트가 즉시 통과했습니다. 이미 구현이 존재하는 것 같습니다.
  TDD 원칙상 이 테스트는 RED 단계로 적합하지 않습니다.
  AC를 더 구체적으로 좁히거나, 다른 동작을 테스트하세요."
  ```

### Step 4: green 단계로 인계

성공적으로 RED 상태 확보 후:

```
RED 단계 완료:
- 테스트: {경로}
- 실패 확인: {메시지}

다음 단계: Skill("oh-my-gx:gx-green") 호출
green 에이전트가 통과를 위한 최소 코드를 작성합니다.
```

오케스트레이터(`oh-my-gx:gx-tdd`)가 호출했으면 자동으로 green 단계 진입.
단독 호출이면 사용자에게 다음 단계 안내.

## 합리화 격파 표

| 변명 | 반박 |
|------|------|
| "이건 너무 간단해서 테스트 필요 없음" | 단순한 코드도 깨진다. 테스트 30초면 끝 |
| "이미 수동으로 테스트했음" | 수동은 재실행 불가. 자동 테스트만 신뢰 가능 |
| "테스트 나중에 작성해도 같은 효과" | 사후 = "이게 뭐 하는 코드?", 사전 = "이게 뭘 해야 함?" |
| "참조용으로 코드 보관" | 보면 적응한다. 그건 사후 테스트. 삭제는 삭제 |
| "이번 한 번만 예외" | 첫 예외가 규칙이 된다 |
| "프로토타입이니까 TDD 생략" | 프로토타입이 그대로 운영에 간다. 처음부터 TDD |

자세한 격파 표는 `.claude/skills/gx-tdd/references/tdd-iron-law.md` 참조.

## Red Flags

다음 생각이 들면 STOP:
- "코드를 잠깐 봐서 어떻게 테스트할지 알아보자"
- "테스트가 너무 강해서 통과시키기 어려우니 약화"
- "이번엔 테스트 없이 가도 될 듯"

→ 모두 Iron Law 1 위반 신호. 즉시 RED 단계로 복귀.

## 다른 스킬 호출 시 절대 규칙 (Iron Law)

✅ 올바름:
- `Skill("oh-my-gx:gx-green")` — GREEN 단계 인계

❌ 금지:
- `Skill("green")` — 접두사 누락(`oh-my-gx:` 필요)
- `Skill("oh-my-gx:gx-commit")` — RED 단계에서 사이클 외부 스킬 호출 금지

**위반 시**: 즉시 중단하고 fully-qualified 이름으로 재호출.
