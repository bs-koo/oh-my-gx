# phase-requirements: PRD Q&A 사이클

**사용자가 "승인"할 때까지 반복한다.** 반복 횟수 제한 없음.

## Hotfix 모드 분기

`--hotfix` 모드이면 경량 PRD를 작성한다:
- product-owner에게 "경량 PRD 작성"으로 동작할 것을 지시한다.
- 포함 섹션: 배경 + 요구사항 + 수용 기준만 (3관점 품질 검증, Q&A 생략).
- 작성 완료 후 사용자에게 전문 표시 + 승인 확인.
- 승인 → `${DEV_DIR}/prd.md`에 저장 후 다음 Phase로 진행.
- 수정 요청 → 1회 수정 후 저장.

hotfix가 아닌 경우 아래 정상 플로우를 따른다.

---

**Step 1**: product-owner agent를 호출한다 (PRD 작성).
`Task(subagent_type="oh-my-gx:product-owner")` — prompt에 다음을 포함:
- 기능/버그 설명: ARGS[0]
- 코드 맵 (phase-setup에서 생성한 초기 맵)
- 프로젝트 타입, 디렉토리 구조
- 프로젝트 루트 경로
- "PRD 작성"으로 동작할 것
- 이전 Q&A 히스토리 (사용자 수정 요청이 있었으면: 이전 PRD 초안 + 사용자 답변)
- PRD 품질 자가 검증 3관점:
  1. **유저 경험 검증**: 이 정책대로 만들면 사용자가 자연스럽게 이해하고 행동할 수 있는가. 혼란을 겪을 수 있는 상태 전환, 빈 화면, 오류 상황이 정의되어 있는가.
  2. **해석 여지 제거**: 개발자·디자이너·PO가 같은 문서를 보고 다르게 해석할 여지가 없는가. "크게", "적절히" 같은 상대적 표현 대신 구체적 수치가 있는가.
  3. **엣지케이스 커버리지**: 빈 상태, 로딩, 에러, 수량/길이의 최솟값·최댓값이 정의되어 있는가. 암묵적으로 처리되는 케이스가 없는가.

**Step 2**: Agent 출력(PRD + 질문)을 사용자에게 **전문 표시**한다. Q&A 여부와 무관하게 항상 전문을 표시한다 (사용자가 PRD를 검토할 수 있도록).

**Step 3**: Agent 출력에서 "탐색 추가 항목"을 파싱하여 코드 맵에 누적한다.

**Step 4**: 질문 여부를 확인한다.

**질문이 있으면** ("추가 확인 사항 없음"이 포함되지 않은 경우):
- PRD를 사용자에게 출력한다.
- Agent 출력의 "확인이 필요한 사항"을 **질문 큐**로 구성한다.
- **순차 질문 루프**를 실행한다 (큐가 비워질 때까지):
  1. 큐에서 질문 1개를 꺼내 `AskUserQuestion(questions: [질문 1개])`로 제시한다. **모든 질문에 권장 답변(현재 맥락에서 가장 가능성 높은 답 또는 합리적 추정)을 options 첫 번째 항목에 `(Recommended)` 라벨로 반드시 제시한다.** 권장 답변 산출이 어려우면 발상 기준점을 주기 위해 예시 후보 1개를 `{ label: "예: ..." }` 형식으로 대체 제시한다. AskUserQuestion 포맷은 **SKILL.md 공유 규칙**의 `에이전트 질문 → AskUserQuestion 변환 규칙`을 따른다.
  2. 답변을 평가한다:
     - **명확하고 충분**: 큐의 다음 항목으로 이동
     - **모호하거나 불완전**: 파고들기 질문을 생성하여 큐 앞에 삽입 후 재질문 (아래 예시 참고)
     - **결정 의존성 발견**: 이 답변이 큐의 후속 항목의 전제 또는 형태를 바꾸면, 영향받는 후속 항목을 재구성하거나 종속 질문을 새로 삽입한다.
     - **코드베이스로 답 가능**: 직접 탐색하여 확인 후 큐의 다음 항목으로 이동
  3. 큐가 비워지면 **공유 이해 확인(align)** 단계를 거친다. 수렴된 답변을 "Q1, Q2, ..." 번호 + "Q → A" 형식으로 요약하여 다음과 같이 확인한다:
     ```
     AskUserQuestion(
       questions: [{
         question: "정리된 답변을 product-owner에게 전달하기 전에 확인해주세요.\n\nQ1: ... → A: ...\nQ2: ... → A: ...",
         header: "정리 확인",
         options: [
           { label: "맞습니다 (Recommended)", description: "이대로 product-owner에 전달" },
           { label: "수정 필요", description: "Other로 이동해서 수정할 질문 번호와 새 답변을 입력해주세요 (예: 'Q3: JWT 대신 세션 방식')" }
         ],
         multiSelect: false
       }]
     )
     ```
     "맞습니다"면 product-owner에게 전달 → Step 2로 돌아간다. "수정 필요"면 사용자가 Other에 입력한 `Q번호: 새 답변` 패턴을 파싱하여 해당 질문을 큐에 재삽입한 뒤 루프 재진입한다. 입력이 패턴과 일치하지 않거나 번호를 식별할 수 없으면, 어떤 항목을 수정할지 선택형 질문으로 다시 묻는다.

**모호함 판단 기준**:
- 정량 수치 없이 "많다", "자주", "크다"만 있는 경우
- 대상이 불명확한 경우 ("사용자들", "팀원들")
- "효율화", "개선", "자동화" 등 추상어만 있는 경우

**파고들기 예시 (PRD 단계)** — 파고들기 질문에도 동일 규칙(권장 답변 + `(Recommended)` 라벨 + Other)이 적용된다. 아래는 질문 텍스트 예시:
- 사용자 규모가 모호하면: "초기 사용자 N명 기준인가요, 1년 후 예상 K명 기준인가요?"
- 비즈니스 목표가 추상적이면: "이 기능으로 어떤 지표(이탈률, 가입률, 처리 시간 등)가 얼마나 개선되길 기대하나요?"
- 대상이 모호하면: "어떤 역할의 사용자가 1차 타겟인가요? (관리자 / 일반 / 특정 권한)"
- 정책이 모호하면: "예외 케이스 X가 발생하면 어떻게 처리하나요? (차단 / 경고 후 진행 / 무시)"

**질문이 없으면** ("추가 확인 사항 없음. PRD가 확정되었습니다."):
- **승인/수정 공통 패턴** (SKILL.md 공유 규칙)에 따라 AskUserQuestion을 사용한다:
  ```
  AskUserQuestion(
    questions: [{
      question: "PRD를 확인해주세요.",
      header: "PRD 확인",
      options: [
        { label: "승인", description: "설계 단계로 진행" },
        { label: "수정 요청", description: "Other로 이동해서 수정할 사항을 자연어로 입력해주세요" }
      ],
      multiSelect: false
    }]
  )
  ```
- 승인 → 다음 Phase로 진행.
- 수정 요청 또는 Other → 입력된 수정 내용을 반영하여 product-owner를 다시 호출 → Step 2로 돌아가 반복한다.

**Phase 완료 후 저장**:
1. `mkdir -p ${DEV_DIR}`로 디렉토리 존재를 보장한다.
2. 확정된 PRD를 `${DEV_DIR}/prd.md`에 Write한다.

**Phase 완료 보고 (요약 모드)**:
PRD 저장 후 사용자에게 **요약만** 출력한다 (Step 2에서 이미 전문을 표시했으므로 반복하지 않음):
```
PRD 확정: <제목>
- 핵심 요구사항: [Must] N건, [Should] N건, [Could] N건
- 수용 기준: N건
- 전문 확인: ${DEV_DIR}/prd.md
```
이후 Phase에서 PRD가 필요하면 파일을 Read하여 Agent prompt에 포함한다.
