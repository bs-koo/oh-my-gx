# gx-green

> **이 스킬**: green — GREEN 단계 (통과를 위한 최소 코드)
> **호출 시 주의**: 이 스킬 내에서 다른 스킬을 호출할 때 반드시 `oh-my-gx:` 접두사를 사용한다.

RED 단계에서 작성된 실패 테스트를 통과시키는 **최소 코드**를 작성한다. 과잉 구현 절대 금지.

---

## Iron Law

```
NO MORE CODE THAN NECESSARY TO PASS THE TEST
```

위반 시:
- 추가된 불필요한 코드 즉시 삭제
- 테스트 통과만을 위한 최소 코드로 재작성
- "어차피 나중에 필요" 금지. YAGNI 우선

---

## 사용 시점

- 보조 스킬 체이닝: `oh-my-gx:gx-red` 완료 직후 자동 호출 (gx-tdd 파이프라인은 이 스킬 대신 green-coder 에이전트를 직접 디스패치)
- RED 선행 없이 단독 호출 금지 (RED 상태가 이미 있으면 단독 호출 가능 — skill-routing의 'GREEN 단계' 단독 호출은 이 경우를 가리킨다)
- 호출 시 RED 상태 검증:
  ```
  실패하는 테스트가 있는가? Bash로 테스트 실행하여 확인.
  없으면: "GREEN 단계는 RED 상태가 선행되어야 합니다. oh-my-gx:gx-red 먼저 호출하세요." 출력 후 중단.
  ```

## 실행 절차

### Step 0: 정체성 확인

```
oh-my-gx:gx-green — GREEN 단계 진입.
실패 테스트를 통과시키는 최소 코드만 작성합니다.
과잉 구현 절대 금지. YAGNI 우선.
```

### Step 1: RED 상태 검증

```bash
# 실패 테스트 명령 재실행
./gradlew test --tests <테스트>  # 또는 npm test -- <테스트>
```

- ✅ 실패하면 → 정상. `git hash-object {테스트 파일}`로 해시를 기록해두고 (Step 3 무결성 비교 기준) Step 2 진행
- ❌ 통과하면 → "이미 구현이 있습니다. RED 단계로 복귀하세요." 중단

### Step 2: 최소 코드 작성 (green-coder 디스패치)

격리된 서브에이전트 호출:

```
Task(subagent_type="oh-my-gx:green-coder"):
  description: "Write minimum code to pass: {test}"
  prompt: |
    당신은 GREEN 단계 최소 코드 작성 전담자입니다.

    [중요] 다음 규칙을 절대 준수합니다:
    1. 실패 테스트 1개만 통과시키는 최소 코드만 작성합니다.
    2. 추가 기능, 에러 핸들링, 검증, 로깅을 미리 넣지 않습니다.
    3. "어차피 나중에 필요"는 YAGNI 위반. 지금 필요한 것만.
    4. 테스트 파일을 수정하지 않습니다. 테스트가 실패하면 코드를 고치고, 테스트를 고치지 않습니다. 테스트 결함이 의심되면 수정하지 말고 "테스트 결함 의심"으로 보고합니다.

    실패 테스트:
    - 파일: {경로}
    - 코드: {테스트 코드}
    - 실패 메시지: {메시지}

    설계서 인터페이스: {대상 컴포넌트의 시그니처}
    프로젝트 루트: {절대 경로 — git rev-parse --show-toplevel, svn은 wc-root, 실패 시 pwd. 파일 도구의 기준점}

    작업:
    1. 테스트를 통과시키는 가장 단순한 구현 작성
    2. 테스트 명령 실행으로 통과 확인
    3. 기존 다른 테스트가 깨지지 않는지 확인
    4. 구현 파일 경로, 코드, 통과 메시지를 보고

    출력 형식:
    - 구현 파일: {경로}
    - 구현 코드: {코드}
    - 통과 확인 명령: {명령}
    - 통과 메시지: {메시지}
    - 다른 테스트 영향: {0건 또는 영향 받은 테스트 목록}
    - 테스트 결함 의심: {없음 | 사유}
```

### Step 3: 통과 확인 (verification)

green-coder 보고 후 직접 검증:

1. **테스트 결함 의심 확인**: green-coder가 "테스트 결함 의심"을 보고했으면 (해시 일치 여부와 무관) → `oh-my-gx:gx-red`로 복귀하여 테스트를 재작성한다
2. **테스트 무결성 확인**: `git hash-object "{테스트 파일}"`을 재실행하여 Step 1 해시와 비교. 불일치(무단 수정) 시 테스트를 **RED 단계에서 인계받은 테스트 코드**로 원복하고 green-coder 재호출 1회. 재차 위반 시 사용자에게 보고
3. 대상 테스트 명령 실행 → **통과** 확인
4. **다른 테스트 영향 없음** 확인 (전체 테스트 실행)
5. **과잉 구현 감지**:
   - 추가된 메서드/필드 중 테스트에서 사용 안 한 것 → 삭제 권고
   - 미리 추가된 에러 핸들링 → 다음 RED 단계로 미루기

**Iron Law 위반 감지**:
- 테스트가 여전히 실패 → green-coder 재호출 (오류 메시지 전달, 최대 2회. 초과 시 중단하고 사용자에게 보고)
- 다른 테스트가 깨짐 → 변경 범위 너무 넓음. 좁혀서 재시도
- 과잉 구현 발견 → 사용자에게 보고 후 정리:
  ```
  "과잉 구현 감지: {불필요한 코드 N줄}.
  YAGNI 원칙상 다음 RED 단계로 미루는 것이 권장됩니다.
  지금 정리할까요?"
  ```

### Step 4: refactor 단계로 인계

성공적으로 GREEN 상태 확보 후:

```
GREEN 단계 완료:
- 구현: {경로}
- 통과 확인: 1/1 pass
- 다른 테스트 영향: 0건

다음 단계: Skill("oh-my-gx:gx-refactor") 호출
refactor 에이전트가 중복 제거 및 정리를 수행합니다.
```

## 합리화 격파 표

| 변명 | 반박 |
|------|------|
| "어차피 나중에 필요할 에러 핸들링" | 지금 필요한 것만. 다음 RED가 그걸 강제할 때 추가 |
| "코드가 너무 단순해서 부끄러움" | 단순함이 목표. 정교함은 REFACTOR에서 |
| "리뷰어가 보기에 불완전해 보임" | RED가 강제하지 않은 건 코드가 아닌 추측 |
| "한 번에 좀 더 만들면 효율적" | 추측 코드는 거의 항상 잘못된 방향 |
| "예외 케이스 미리 처리" | 예외도 RED 단계로 강제하라 |

---

## Red Flags

- "이 정도는 미리 처리해도 될 듯"
- "에러 메시지를 좀 더 친절하게"
- "관련 메서드 하나만 추가"
- "이왕 만드는 김에 X도"

→ 모두 YAGNI 위반 신호. 추가 코드 삭제.

## 다른 스킬 호출 시 절대 규칙 (Iron Law)

✅ 올바름:
- `Skill("oh-my-gx:gx-refactor")` — REFACTOR 단계 인계
- `Skill("oh-my-gx:gx-red")` — "테스트 결함 의심" 보고 시 RED 재작성 복귀

❌ 금지:
- `Skill("refactor")` — 일관성 위반
- `Skill("oh-my-gx:gx-commit")` — GREEN 단계에서 사이클 외부 스킬 호출 금지

**위반 시**: 즉시 중단하고 fully-qualified 이름으로 재호출.
