# oh-my-gx

**기획서를 주면 AI 에이전트 팀이 PRD, 설계, 구현, 리뷰, PR까지 처리합니다.**

Java Spring Boot / 풀스택(Spring Boot + Vue/React) 프로젝트용 Claude Code 플러그인

## 사용법

명령어 문법을 외울 필요 없습니다. 자연어로 말하면 AI가 의도를 파악하고 적절한 스킬과 모드를 자동 선택합니다.

### 기획서 기반 개발 (가장 일반적인 시나리오)

`context` → `lens`(선택) → `dev` → `humanizer`(선택) 순서로 진행합니다.

**1단계: context — 기획서로 도메인 지식 등록**

```
사용자: requirements 폴더에 사용량 분석 기획서가 있어. 이걸 보고 context 만들어줘

AI: 문서를 분석했습니다.
    - 도메인명: 사용량 분석
    - 요구사항: 9건
    - 용어: 6건 (일일 활성 사용자, 세션, 리텐션율 등)

    몇 가지 확인이 필요합니다.
    1. 이 도메인의 담당 PM/PO는 누구인가요?
    2. 관련 레포가 있나요?

사용자: PM은 박지영, 레포는 이것만 써

AI: context/사용량-분석/ 디렉토리를 생성했습니다. (README.md, glossary.md, architecture.md 등)
```

**2단계: lens — 현행 정책 파악 (선택)**

기존 코드에 관련 로직이 있을 때, 변경 전 현황을 파악합니다.

```
사용자: 현재 사용자 활동 로깅이 어떻게 되어 있는지 정리해줘

AI: [정책 보고서]
    - 로그인/로그아웃 시점에 ActivityLog 테이블에 기록
    - 페이지 뷰는 수집하지 않음
    - 보관 기간: 90일 (ActivityLogCleanupScheduler에서 확인)

    변경 아이디어가 있으면 알려주세요. 영향도를 분석해 드립니다.
```

**3단계: dev — 개발 요청**

```
사용자: 사용량 분석 대시보드 기능 개발해줘

AI(PO): PRD를 작성하기 전에 확인이 필요합니다.
    1. 분석 대상 지표는 무엇인가요?
       (a) DAU, 세션 수, 평균 체류시간 (권장)
       (b) 위 항목 + 페이지별 방문 수
       (c) 직접 입력
    2. 조회 기간 범위는?

사용자: b, 최근 30일

AI(PO): PRD를 작성했습니다. 확인해주세요.
    → [PRD 내용 표시]

사용자: 승인

AI(설계자): 기술 설계를 완료했습니다.
    → [변경 범위, API, DB 스키마, 구현 순서 등 표시]

사용자: 승인

AI(개발자): 구현 완료. [1/4] UsageStatService... [2/4] UsageStatController... [3/4] 집계 스케줄러... [4/4] 테스트...
AI(리뷰어): 코드 리뷰 + 보안 감사 완료. Critical 이슈 없음.
AI: PR을 생성했습니다. → https://github.com/...
```

**4단계: humanizer — 산출물 교정 (선택)**

PRD나 설계 문서에 AI가 쓴 티가 나면 교정합니다.

```
사용자: .dev/prd.md AI 글쓰기 흔적 교정해줘

AI: 감지 리포트 — 총 8건 (P1: 3건, P2: 5건)
    수정된 문서를 확인해주세요.
```

매 단계마다 AI가 질문하고, 사용자가 답변하거나 승인해야 다음으로 넘어갑니다.

## 개발 파이프라인

`context` → `lens`(선택) → `dev` → `humanizer`(선택) 순서로 사용하면 가장 정확한 결과를 얻습니다. `dev`만 단독으로 써도 됩니다.

| 단계 | 역할 | 필수 |
|------|------|------|
| **context** | 기획서/코드베이스 기반 도메인 지식 등록 | 권장 |
| **lens** | 현행 코드의 비즈니스 정책 분석 (읽기 전용) | 선택 |
| **dev** | PRD → 설계 → 구현 → 리뷰 → 커밋/PR | 필수 |
| **humanizer** | PRD/설계 문서의 AI 글쓰기 흔적 교정 | 선택 |

### context

기획서, 요구사항 문서, 코드베이스를 기반으로 도메인 지식을 등록합니다. 상황에 따라 모드를 자동 판단합니다.

```
"requirements 폴더에 있는 기획서들 보고 context 만들어줘"        ← 문서 기반 생성
"사용량 분석 도메인 등록해줘"                                  ← Q&A 기반 생성
"코드베이스 분석해서 context 자동 생성해줘"                     ← 코드 스캔
"사용량 분석 도메인 동기화해줘"                                ← git 히스토리 기반 진행도 갱신
```

등록된 context는 `dev` 실행 시 자동으로 참조됩니다. 용어, 아키텍처, 담당자, 구현 진행도(`status.md`)를 관리합니다.

### lens

코드에서 비즈니스 정책을 찾아 PO/PD가 이해할 수 있는 보고서로 제공합니다. 코드를 수정하지 않습니다.

```
"현재 사용자 활동 로깅이 어떻게 되어 있는지 정리해줘"
"로그 보관 기간을 180일로 늘리면 어디에 영향이 가?"
```

정책 보고서 출력 후, 변경 아이디어를 입력하면 복잡도와 리스크 분석까지 이어집니다.

### dev

자연어 요청 하나로 PRD 작성부터 PR 생성까지 전체 사이클을 수행합니다. 요청의 의도를 자동으로 파악합니다.

```
"사용량 분석 대시보드 기능 개발해줘"         ← 전체 사이클
"집계 스케줄러 오류 긴급 수정해줘"          ← hotfix 모드 (설계/리뷰 생략)
"PRD만 작성해줘"                          ← 특정 단계만 실행
"아까 하던 작업 이어서 해줘"               ← 중단된 단계부터 재개
"지금 어디까지 됐어?"                      ← 진행 상태 확인
```

**에이전트 팀 구성:**

| 에이전트 | 역할 | 단계 |
|---------|------|------|
| 제품책임자(PO) | 요구사항 구체화, PRD 작성, 인수 검증 | requirements, complete |
| 설계자 | 기술 설계 (변경 범위, API, 구현 순서) | design |
| 설계 비판자 | 암묵적 가정 도전, 과잉 설계 식별 | design (중형 이상) |
| 개발자 | 설계 기반 코드 구현 | implement |
| QA 매니저 | 코드 리뷰 + 스펙 충족 검증 | implement, review |
| 보안 감사자 | 정책/보안/허점 교차 검증 | review |

## 보조 기능

```
"커밋해줘"          → 브랜치명에서 타입 파싱, 변경사항 분석, 한국어 커밋 메시지 생성
"PR 만들어줘"       → 커밋 히스토리 분석, PR 제목/본문 자동 생성
```

## 안전장치

- PR 생성까지만 자동화합니다. **PR 머지는 사용자가 직접** 수행합니다.
- `git push --force`, `gh pr merge`는 설정 수준에서 차단됩니다.
- 보호 브랜치(main)에서 직접 커밋을 차단합니다.
- 커밋 전 민감 파일(`.env`, `*.key`, `*.pem`, `credentials*`, `*secret*`) 감지 시 경고합니다.
- 빌드 아티팩트(`build/`, `node_modules/` 등)가 tracked 상태이면 자동으로 `.gitignore` 보강을 제안합니다.

## FAQ

<details>
<summary><b>context 없이도 dev를 실행할 수 있나요?</b></summary>

네. 없어도 동작합니다. 다만 context를 등록하면 AI가 도메인 용어를 정확히 이해하여 더 정확한 코드를 생성합니다.
</details>

<details>
<summary><b>dev를 실행하면 바로 코드를 짜나요?</b></summary>

아닙니다. PO 에이전트가 먼저 Q&A로 요구사항을 구체화하고(PRD), 설계자가 기술 설계를 거친 뒤 사용자가 승인해야 구현에 들어갑니다. 각 단계마다 선택형/자유입력형 질문으로 확인을 받습니다.
</details>

<details>
<summary><b>dev 도중에 멈추면 처음부터 다시 해야 하나요?</b></summary>

아닙니다. "이어서 해줘"라고 말하면 `.dev/state.md`에 저장된 진행 상태를 기반으로 중단된 단계부터 재개합니다.
</details>

<details>
<summary><b>PR이 자동으로 머지되나요?</b></summary>

아닙니다. PR 생성까지만 자동화합니다. `gh pr merge`는 설정 수준에서 차단되어 있습니다.
</details>

<details>
<summary><b>플러그인 업데이트는?</b></summary>

`/plugin marketplace update oh-my-gx`
</details>
