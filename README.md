## 사용법

자연어로 말하면 의도에 맞는 스킬이 발동됩니다. 명령어를 외울 필요 없습니다.

| 이렇게 말하면 | 발동 스킬 |
|--------------|----------|
| "기획서 보고 context 만들어줘" | context |
| "현재 로깅 정책 정리해줘" | lens |
| "대시보드 기능 개발해줘" | dev |
| "긴급 수정해줘" | dev (hotfix) |
| "PRD만 작성해줘" | dev (단일 단계) |
| "이어서 해줘" | dev (재개) |
| ".dev/prd.md AI 흔적 교정해줘" | humanizer |
| "커밋해줘" | commit |
| "PR 만들어줘" | pull-request |

### 개발 흐름

`context` → `dev` 두 단계로 개발합니다. `dev`만 단독으로 써도 됩니다.

1. `requirements/` 폴더에 기획서(PDF, 이미지, 텍스트)를 넣습니다
2. "context 만들어줘"로 도메인 지식을 등록합니다
3. "개발해줘"로 PRD → 설계 → 구현 → 리뷰 → PR까지 실행합니다

각 단계 사이에 사용자 승인이 필요합니다. 승인 없이 다음으로 넘어가지 않습니다.

## context — 도메인 지식 관리

기획서, 요구사항 문서, 코드베이스를 분석하여 도메인 지식을 `context/{도메인}/`에 등록합니다. 등록된 context는 `dev` 실행 시 자동 참조됩니다.

### 자연어 의도 파싱

```
"requirements 폴더에 있는 기획서 보고 context 만들어줘"   ← 문서 기반 생성
"사용량 분석 도메인 등록해줘"                            ← Q&A 기반 생성
"코드베이스 분석해서 context 자동 생성해줘"               ← 코드 스캔
"사용량 분석 도메인 동기화해줘"                          ← git 히스토리 기반 진행도 갱신
```

### 5가지 모드와 자동 판단 로직

context는 인자와 현재 상태를 조합하여 자동으로 모드를 결정합니다.

| 조건 | 모드 | 동작 |
|------|------|------|
| `context/` 없음 + 인자 없음 | **스캔** | 코드베이스의 패키지 구조, 엔티티, API 엔드포인트를 분석하여 도메인 구조 자동 생성 |
| `context/` 없음 + 도메인명 지정 | **스캔+신규** | 코드 스캔 후 해당 도메인 우선 생성 |
| `context/` 있음 + 도메인명 | **신규** | Q&A 5개 질문(문제, 현재 프로세스, 안 하면, 사용자/규모, 담당)으로 도메인 생성 |
| `--from <파일>` 지정 | **문서 기반** | PDF/이미지/텍스트를 읽고 내용 추출 → context 생성 |
| 기존 도메인 + `--sync` | **동기화** | git log + PR 히스토리 분석 → status.md 자동 갱신 |

### 프로세스 흐름 — 스캔 모드

```
패키지/디렉토리 구조 스캔
        ↓
엔티티/모델 파일 탐색 (병렬)
API 엔드포인트 탐색 (병렬)
설정 파일 읽기 (병렬)
        ↓
도메인 분류 (패키지/엔티티/API 기반 추론)
        ↓
초안 생성 (README, PROJECTS, glossary, architecture, status)
        ↓
사용자 검토 + 도메인 추가/제거/이름 변경
        ↓
확정 및 인덱스 생성
```

### 프로세스 흐름 — 신규 (Q&A) 모드

```
도메인명 확인
    ↓
검증 질문 5개 (문제, 현재 프로세스, 안 하면, 사용자/규모, 담당)
    ↓
답변 품질 판단 (3/5 구체적이면 통과, 아니면 심화 질문 1라운드)
    ↓
관련 프로젝트(GHE 레포) 확인
    ↓
디렉토리 및 문서 생성
    ↓
인덱스 업데이트 + 완료 안내
```

### 생성되는 문서 구조

```
context/{도메인}/
├── README.md          ← 도메인 개요 (배경, 문제, 사용자, 성공기준, 담당자)
├── PROJECTS.md        ← 관련 GHE 레포 매핑
├── glossary.md        ← 도메인 용어 사전
├── architecture.md    ← 전체 구조 요약 + 주제 문서 링크
├── status.md          ← 구현 추적 (AC별 ✅/⬜)
└── {주제}/README.md   ← 주제별 상세 정책/설계
```

### 답변 품질 게이트

context는 모호한 답변을 허용하지 않습니다.

- "많다", "자주", "크다"만 있는 답변 → 정량 수치를 요구합니다
- "효율화", "개선" 등 추상어만 있는 답변 → 구체화를 요청합니다
- 그래도 모르는 항목은 ❓로 남기고 진행합니다

## dev — 전체 개발 파이프라인

자연어 요청 하나로 PRD 작성부터 PR 생성까지 전체 사이클을 수행합니다.

### 자연어 의도 파싱

dev는 자연어를 분석하여 자동으로 모드를 결정합니다.

| 감지 패턴 | 모드 | 예시 |
|-----------|------|------|
| `상태`, `진행`, `어디까지` | STATUS | "지금 어디까지 됐어?" |
| `이어서`, `계속`, `재개` | RESUME | "아까 하던 작업 이어서 해줘" |
| `긴급`, `핫픽스`, `급한`, `빨리 고쳐` | HOTFIX | "로그인 버그 긴급 수정해줘" |
| `설계만`, `PRD만`, `구현만` | 단일 Phase | "설계만 해줘" |
| `{branch}에서`, `{branch} 기반` | BASE 추출 | "develop 브랜치 기반으로 작업해줘" |
| 위 어디에도 해당 안 됨 | NORMAL | "로그인 기능 추가해줘" |

### 6단계 파이프라인

```
┌─────────────────────────────────────────────────────────────┐
│                    정상 경로 (Normal)                        │
│                                                             │
│  Setup → Requirements → Design → Implement → Review → Complete │
│   환경      PRD Q&A      설계      구현+자기점검   QA+보안   커밋/PR  │
│                                                             │
├─────────────────────────────────────────────────────────────┤
│                    긴급 경로 (Hotfix)                        │
│                                                             │
│  Setup → Requirements(경량) → Implement → Complete           │
│   환경      경량 PRD           구현+자기점검   커밋/PR         │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 9개 에이전트 팀

dev 스킬의 핵심은 역할이 분리된 에이전트 팀이 협업하는 구조입니다. 각 에이전트는 고유한 관점과 역할을 가지며, 적합한 모델이 배정됩니다.

| 에이전트 | 관점 | 역할 | 모델 | 활동 Phase |
|---------|------|------|------|-----------|
| **product-owner** | "뭘 만들지" | PRD 작성, 인수 검증 | Sonnet | requirements, complete |
| **architect** | "어떻게 만들지" | 기술 설계 (변경 범위, API, 구현 순서) | Opus | design |
| **design-critic** | "이 가정이 맞나" | 암묵적 가정 도전, 과잉 설계 식별 | Opus | design (중형 이상만) |
| **coder** | "만든다" | 설계 기반 코드 구현 + 버그 수정 | Opus | implement, review(수정) |
| **qa-manager** | "스펙대로 됐나" | 코드 리뷰, 스펙 충족 검증 | Sonnet | implement(자기점검), review |
| **security-auditor** | "뭘 놓쳤나" | 정책/보안/허점 교차 검증 | Sonnet | review |
| **researcher** | "이해한다" | 코드베이스 조사, 기술 비교 | Sonnet | (독립 호출) |
| **hacker** | "다른 길이 있다" | 제약 우회, 정체 탈출 | Sonnet | (정체 감지 시) |
| **simplifier** | "더 작게 만들자" | 복잡도 제거, 범위 축소 | Sonnet | (정체 감지 시) |

#### 모델 라우팅 원칙

모든 에이전트가 같은 모델을 쓰지 않습니다. 작업 특성에 따라 모델을 배정합니다.

- **Opus** (비용 높음, 추론 깊음): 비판적 분석, 구조적 설계, 코드 구현
- **Sonnet** (비용 효율): PRD/리뷰 산출물 생성, 정체 탈출 빠른 판단

### 각 Phase 상세

#### Phase 1: Setup — 작업환경 준비

```
진행 중 작업 감지 (.dev/state.md)
    ↓ (이전 작업 있으면 재개 여부 확인)
Git 저장소 확인
    ↓
베이스 브랜치 결정 + 동기화 (git pull)
    ↓
프로젝트 정보 수집 (4개 작업 병렬):
  ├── 프로젝트 타입 감지 (build.gradle.kts → java-spring 등)
  ├── 디렉토리 구조 수집
  ├── CLAUDE.md 컨벤션 확보
  └── 도메인 컨텍스트 매칭 (context/*/PROJECTS.md)
    ↓
관련 코드 맵 생성 (키워드 추출 → Grep → 핵심 파일 15개 이내)
    ↓
작업 브랜치 생성 (이슈 키 또는 영문 키워드)
    ↓
.gitignore 자동 보강 + 진행 상태 초기화
```

#### Phase 2: Requirements — PRD Q&A

```
product-owner 에이전트 호출 (PRD 작성)
    ↓
PRD 전문 표시 + 3관점 품질 자가 검증:
  ├── 유저 경험 검증: 사용자가 자연스럽게 이해하고 행동할 수 있는가
  ├── 해석 여지 제거: "크게", "적절히" 대신 구체적 수치가 있는가
  └── 엣지케이스 커버리지: 빈 상태, 로딩, 에러, 최솟값/최댓값이 정의되었는가
    ↓
질문이 있으면 → AskUserQuestion으로 변환 → 답변 반영 (최대 1회)
    ↓
사용자 승인 (승인 / 수정 요청)
    ↓
.dev/prd.md에 저장
```

#### Phase 3: Design — 설계 Q&A

```
architect 에이전트 호출 (기술 설계)
    ↓
설계 초안 전문 표시
    ↓
설계 비판 검토 (중형/대형만):
  └── design-critic이 암묵적 가정 도전, 과잉 설계 식별
      ├── MUST-ADDRESS → architect 설계 수정
      └── CONSIDER → 참고 사항으로 기록
    ↓
질문이 있으면 → AskUserQuestion (최대 2회 반복)
    ↓
사용자 승인 (승인 / 수정 요청)
    ↓
.dev/design.md에 저장
```

#### Phase 4: Implement — 구현 + 자기점검

```
구현 계획 제시 (변경 파일, 변경 유형, 구현 순서)
    ↓
사용자 승인
    ↓
coder 에이전트 구현 (설계서 순서대로)
    ↓
구현 완료 요약 보고 ([1/M] 파일 - 변경 요약)
    ↓
자기점검 (qa-manager):
  ├── Critical → coder 자동 수정 (1회)
  ├── Warning/Info → phase-review로 이월
  └── QUESTION → phase-review로 이월
    ↓
.dev/self-check.md에 저장
```

#### Phase 5: Review — QA + 보안 감사

```
Mechanical Gate (빌드 + 테스트):
  ├── 빌드 실패 → coder 자동 수정 → 1회 재시도
  └── 테스트 실패 → coder 자동 수정 → 1회 재시도
    ↓
qa-manager + security-auditor 병렬 호출:
  ├── QA: 코드 리뷰 + 스펙 충족 검증
  │   ├── CERTAIN: 확실한 문제 → 자동 수정
  │   └── QUESTION: 확인 필요 → 사용자에게 질문
  └── 종합 검증 (security-auditor): PRD + 설계서 + diff 교차 검증
      ├── CRITICAL/HIGH → 수정 필요
      └── MEDIUM → 사용자 판단
    ↓
결과 합산 → Trust Ledger 생성 (.dev/trust-ledger.md)
    ↓
Critical → coder 자동 수정 → 재리뷰 (최대 2회)
```

#### Phase 6: Complete — 완료

```
인수 검증 (product-owner):
  └── PRD 수용 기준 대비 구현 결과 검증
      ├── ACCEPT → 다음 단계
      └── REJECT → coder 수정 → 인수 검증 재실행 (1회)
    ↓
/commit 스킬 실행 (빌드 체크 + 한국어 커밋)
    ↓
/pull-request 스킬 실행:
  ├── PR 본문에 비즈니스 맥락 + Trust Ledger 요약 포함
  └── Google Chat 알림 전송
    ↓
도메인 status.md 갱신 (⬜→✅)
    ↓
context 환류 제안 (glossary, 주제 문서, architecture)
```

### 정체 감지 + 에스컬레이션

구현과 리뷰 단계에서 같은 에러가 반복되거나 진전이 없으면, 자동으로 전문 에이전트에게 에스컬레이션합니다.

| 감지 패턴 | 상황 | 1차 대응 | 2차 대응 |
|----------|------|---------|---------|
| SPINNING | 동일 에러 2회 연속 | hacker에 제약 우회 분석 | researcher에 근본 원인 분석 |
| OSCILLATION | 접근법 A→B→A 왕복 | architect에 설계 재검토 | 사용자에게 두 접근법 제시 |
| NO_DRIFT | 코드 변경 없이 반복 | hacker에 우회 경로 요청 | researcher에 코드베이스 탐색 |
| DIMINISHING_RETURNS | 수정은 줄고 결과 안 나옴 | simplifier에 범위 축소 | 사용자에게 방향 전환 확인 |

### 컨텍스트 경량화 전략

에이전트에게 모든 정보를 전달하면 토큰 낭비입니다. 역할별로 필요한 섹션만 슬라이싱합니다.

| 에이전트 | 받는 정보 | 받지 않는 정보 |
|---------|----------|--------------|
| product-owner (PRD) | ARGS + 코드 맵 + 프로젝트 구조 | 설계서, diff |
| architect (설계) | PRD 전체 + 코드 맵 + 컨벤션 | diff |
| coder (구현) | 설계서 전체 + 코드 맵 | PRD (설계서에 반영됨) |
| qa-manager (자기점검) | PRD 요구사항+수용기준만 | 설계서, 배경 |
| security-auditor | PRD + 설계서 + diff + 코드 맵 | (전체 접근) |

### 상태 저장과 재개

dev는 `.dev/state.md`에 현재 진행 상태를 기록합니다. "이어서 해줘"라고 말하면 중단된 Phase의 중단된 Step부터 정확히 재개합니다.

## lens — 비즈니스 정책 탐지

코드에서 비즈니스 정책을 찾아 PO/PD가 읽을 수 있는 보고서로 출력합니다. **읽기 전용** — 코드를 수정하지 않습니다.

### 자연어 의도 파싱

```
"현재 사용자 활동 로깅이 어떻게 되어 있는지 정리해줘"
"로그 보관 기간을 180일로 늘리면 어디에 영향이 가?"
"결제 한도 정책 확인해줘 --detail"
```

### 5단계 Phase

```
Prepare: 쿼리 키워드 추출 → 프로젝트 확정
    ↓
Explore: Agent(Explore)로 코드에서 정책 구현 발견
  ├── 1단계: 후보 수집 (Glob + Grep, Read 없이)
  ├── 2단계: 입구 추적 (Controller/Handler Read)
  └── 3단계: 핵심 파일 정독 (Service/Policy/Rule Read)
    ↓
Report: 발견 사항을 "대상에게 무슨 일이 일어나는가" 관점으로 합성
  └── 유사 기능 포함 여부를 사용자에게 확인
    ↓
(선택) Impact: architect + security-auditor 병렬로 복잡도 + 리스크 분석
    ↓
(선택) Impact-Report: PO 친화적 영향도 보고서 생성
```

### 페르소나: 기술 번역자

lens는 코드를 읽되, 출력은 비즈니스 언어로 번역합니다.

- 기술 용어 최소화. 불가피하면 괄호에 비즈니스 용어 병기
  - 예: `PurchaseLimitPolicy` → "구매 한도 정책 (`PurchaseLimitPolicy`)"
- 코드 위치는 역할/도메인을 먼저, 파일명을 괄호에
  - 예: "구매 도메인 서비스 (`RandomBoxService.kt`)"
- 코드 변경을 제안하지 않음. 확인이 필요한 사항만 안내

### 보고서 예시 구조

```
## 정책 탐지 보고서: 결제 한도

### 1. 일일 결제 한도
- 설명: 사용자당 하루 최대 결제 금액을 제한한다
- 핵심 값: DAILY_LIMIT = 1,000,000원
- 규칙: 당일 누적 결제액 + 현재 결제액 > DAILY_LIMIT이면 거부
- 확인 위치: 결제 도메인 서비스 (PaymentLimitPolicy.kt)

### 확인이 필요한 사항
- 한도 초과 시 부분 결제(한도까지만 결제)를 허용할지 정책이 명시되어 있지 않습니다
```

## humanizer — AI 글쓰기 교정

PRD나 설계 문서에서 AI 글쓰기 패턴(40+가지)을 감지하고 교정합니다.

### 두 가지 모드

| 모드 | 동작 | 용도 |
|------|------|------|
| **audit** | 패턴 감지 + 리포트만 출력. 텍스트를 수정하지 않음 | 어디가 문제인지 확인만 하고 싶을 때 |
| **rewrite** | 패턴 감지 + 직접 교정 | AI 흔적을 제거한 최종본이 필요할 때 |

### 프로세스 흐름

```
모드 결정 (audit / rewrite)
    ↓
입력 확인 (텍스트, 파일 경로, 글로브 패턴)
    ↓
콘텐츠 유형 파악:
  ├── 블로그/에세이 → 모든 패턴 + 숨결 주입
  ├── 기술 문서 → 수식어/filler 제거, 감정 주입 금지
  ├── 마케팅/카피 → 과장 줄이되 설득력 유지
  └── 학술/리포트 → 정확성 중심, weasel word 제거
    ↓
패턴 스캔 (40+ 패턴, 심각도 분류):
  ├── P1 (확실한 AI 흔적) → 즉시 수정
  ├── P2 (의심스러운 패턴) → 맥락 판단
  └── P3 (스타일 개선) → 선택적 수정
    ↓
결과 출력 (audit: 리포트 / rewrite: 교정된 텍스트)
```

### 40+ 감지 패턴 카탈로그

#### 한국어 P1 — 확실한 AI 흔적 (즉시 수정)

| 코드 | 패턴 | 감지 키워드 예시 |
|------|------|-----------------|
| K1 | 도입부 상투어 | "오늘날", "알아보겠습니다", "이번 글에서는" |
| K2 | 과장 수식어 | "혁신적인", "체계적인", "다양한"(과용), "탁월한" |

| K4 | "이를 통해" 연쇄 | "이를 통해", "이를 바탕으로", "이를 활용하여" |
| K5 | 무의미한 중요성 강조 | "중요성은 아무리 강조해도", "핵심적인 역할" |
| K10 | 결론부 상투어 | "결론적으로", "발전이 기대됩니다" |

### "숨결 불어넣기" — 블로그/에세이 전용

AI 패턴을 제거하는 것만으로는 부족합니다. 깨끗하지만 무미건조한 글도 AI처럼 보입니다.

블로그/에세이 유형에서는 다음을 추가합니다:
- 의견을 가져라 — "솔직히 이건 좀 애매하다"가 중립 나열보다 낫다
- 리듬을 바꿔라 — 짧은 문장. 그리고 긴 문장. 섞어라
- '나'를 쓸 때는 써라 — 1인칭이 비전문적인 게 아니다
- 약간의 지저분함을 허용하라 — 완벽한 구조는 알고리즘 냄새가 난다

## commit — 커밋 자동화

변경사항을 분석하여 브랜치 타입 기반 한국어 커밋 메시지를 자동 생성합니다.

### 프로세스 흐름

```
사전 확인:
  ├── Git 저장소 확인
  ├── 변경사항 존재 확인
  └── 빌드 실행 (프로젝트 타입별 자동 감지)
    ↓
타입 파싱:
  └── 브랜치명에서 첫 번째 / 앞 세그먼트 추출
      예: feat/login → feat, fix/auth-bug → fix
    ↓
커밋 메시지 생성:
  ├── git diff --stat으로 변경 요약
  ├── 변경 파일 50개 이하면 상세 diff 확인
  └── {type}: 한국어 요약 (50자 이내) + 본문 bullet
    ↓
커밋 실행:
  ├── 빌드 아티팩트 tracking 해제 (build/, node_modules/)
  ├── 민감 파일 감지 (.env, *.key, *.pem, credentials*, *secret*) → 경고
  ├── 20개 초과 변경 → 전체 스테이징 확인
  └── HEREDOC 포맷으로 커밋
    ↓
context 동기화 체크:
  └── 변경 파일이 기존 도메인과 관련 있으면 동기화 제안
```

### 커밋 메시지 예시

```
feat: 로그인 기능 추가

- 로그인 API 엔드포인트 구현
- JWT 토큰 발급 로직 추가
- 로그인 폼 UI 구현
```

## pull-request — PR 자동화

커밋 히스토리를 분석하여 PR 제목과 본문을 자동 생성합니다.

### 프로세스 흐름

```
사전 확인:
  ├── gh CLI 존재 + 인증 확인
  ├── origin remote 확인
  ├── 베이스 브랜치 자동 감지
  └── 미커밋 변경사항 경고
    ↓
타입 파싱:
  └── 브랜치명 → PR 타입 매핑 (feat→FEATURE, fix→BUGFIX 등)
    ↓
PR 제목 생성:
  └── [TYPE] 서술형 한국어 문장 (50자 이내)
      예: [FEATURE] 로그인 기능을 구현한다.
    ↓
PR 본문 생성:
  ├── Background: 왜 이 변경이 필요한가 (문제/맥락)
  ├── Summary: 무엇을 했는가 (핵심 접근 방식)
  ├── Changes: 기능 단위로 무엇이 바뀌었는가
  └── Checklist: 변경 내용에 따라 동적 생성
    ↓
PR 생성 + Google Chat 알림
```

### PR 본문 작성 규칙

- 모든 섹션은 bullet이 아닌 **문장형 서술**로 작성
- Background ≠ Summary: Background는 "왜(문제/맥락)", Summary는 "무엇을(해결책)"
- Changes는 파일명 나열이 아니라 **기능 관점**으로 서술
- Checklist는 변경 내용에 따라 **동적 생성** (테스트 없으면 테스트 항목 생략)

<summary><b>context 없이도 dev를 실행할 수 있나요?</b></summary>

네. 없어도 동작합니다. 다만 context를 등록하면 AI가 도메인 용어를 정확히 이해하여 더 정확한 코드를 생성합니다.
</details>

<details>
<summary><b>dev를 실행하면 바로 코드를 짜나요?</b></summary>

아닙니다. PO 에이전트가 먼저 Q&A로 요구사항을 구체화하고(PRD), 설계자가 기술 설계를 거친 뒤 사용자가 승인해야 구현에 들어갑니다. 각 단계마다 선택형/자유입력형 질문으로 확인을 받습니다.
</details>

<details>
<summary><b>dev 도중에 멈추면 처음부터 다시 해야 하나요?</b></summary>

아닙니다. "이어서 해줘"라고 말하면 `.dev/state.md`에 저장된 진행 상태를 기반으로 중단된 단계부터 재개합니다.
</details>

<details>
<summary><b>PR이 자동으로 머지되나요?</b></summary>

아닙니다. PR 생성까지만 자동화합니다. `gh pr merge`는 설정 수준에서 차단되어 있습니다.
</details>

<details>
<summary><b>플러그인 업데이트는?</b></summary>

`/plugin marketplace update oh-my-gx`
</details>

<details>
<summary><b>에이전트가 모두 같은 AI 모델을 사용하나요?</b></summary>

아닙니다. 작업 특성에 따라 모델을 배정합니다. 비판적 분석(design-critic)과 코드 구현(coder)은 추론이 깊은 Opus를, PRD 생성(product-owner)과 코드 리뷰(qa-manager)는 비용 효율적인 Sonnet을 사용합니다.
</details>

<details>
<summary><b>hotfix 모드는 뭐가 다른가요?</b></summary>

설계(Design)와 리뷰(Review) 단계를 건너뛰지만, 경량 PRD와 인수 검증은 실행합니다. "긴급", "핫픽스", "급한", "빨리 고쳐" 같은 표현을 감지하면 자동으로 hotfix 모드로 전환됩니다.
</details>

<details>
<summary><b>lens는 코드를 수정하나요?</b></summary>

아닙니다. lens는 읽기 전용입니다. 코드에서 비즈니스 정책을 찾아 보고서로 출력만 합니다. Edit, Write 도구를 프로젝트 파일에 사용하지 않습니다.
</details>
