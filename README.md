# oh-my-gx

**GX 사업본부 개발자들을 위한 PRD, 설계, 구현, 리뷰, PR까지 처리하는 개발 자동화 플러그인**

풀스택(Spring Boot + Vue/Nuxt) 프로젝트용 Claude Code 플러그인

## 설치 및 시작

```bash
# Claude Code CLI에서 실행
/plugin marketplace add bs-koo/oh-my-gx
/plugin install oh-my-gx@oh-my-gx

# 플러그인 리로드
/reload-plugins

# 시작
/oh-my-gx:setup
```

## 사용법

자연어로 요청하면 의도에 맞는 스킬이 자동 선택됩니다. 명령어를 외울 필요 없습니다.

### 워크플로우

기획서가 있는 프로젝트라면 `context` → `dev` 순서가 기본입니다. `dev`만 단독으로 써도 됩니다.

**기획서 준비** — 프로젝트 루트에 `requirements/` 폴더를 만들고 요구사항 문서(PDF, 이미지, 텍스트)를 넣어둡니다.

**context — 도메인 지식 등록** — 기획서를 읽고 도메인 용어, 아키텍처, 담당자를 `context/{도메인}/` 디렉토리에 정리합니다. "requirements 폴더 보고 context 만들어줘"처럼 요청하면 문서를 분석한 뒤, 부족한 정보(담당자, 관련 레포 등)를 질문으로 보충합니다.

**lens — 현행 정책 파악 (선택)** — 기존 코드에 관련 로직이 있을 때 변경 전 현황을 파악합니다. 코드를 수정하지 않고, PO/PD가 읽을 수 있는 정책 보고서를 출력합니다. 변경 아이디어를 이어서 입력하면 영향도 분석까지 수행합니다.

**dev — 개발 실행** — 요청 하나로 아래 단계를 순서대로 거칩니다.

1. **requirements** — PO 에이전트가 선택형/자유입력 질문으로 요구사항을 구체화하고 PRD를 작성합니다.
2. **design** — 설계자가 변경 범위, API, DB 스키마, 구현 순서를 정리합니다. 규모가 크면 설계 비판자가 과잉 설계나 숨은 가정을 점검합니다.
3. **implement** — 개발자가 승인된 설계대로 코드를 구현합니다.
4. **review** — QA 매니저가 코드 리뷰와 스펙 충족을 검증하고, 보안 감사자가 취약점을 점검합니다.
5. **complete** — PO가 인수 검증을 마치면 커밋과 PR을 생성합니다.

각 단계 사이에 사용자 승인이 필요합니다. 승인 없이 다음 단계로 넘어가지 않습니다.

"긴급 수정해줘"라고 하면 설계/리뷰를 생략하는 hotfix 모드로, "PRD만 작성해줘"라고 하면 특정 단계만 실행합니다. 중단된 작업은 "이어서 해줘"로 재개됩니다.

**humanizer — 산출물 교정 (선택)** — PRD나 설계 문서에서 AI 글쓰기 패턴을 감지하고 교정합니다.

## 개발 파이프라인

| 단계 | 역할 | 필수 |
|------|------|------|
| **context** | 기획서/코드베이스 기반 도메인 지식 등록 | 권장 |
| **lens** | 현행 코드의 비즈니스 정책 분석 (읽기 전용) | 선택 |
| **dev** | PRD → 설계 → 구현 → 리뷰 → 커밋/PR | 필수 |
| **humanizer** | PRD/설계 문서의 AI 글쓰기 흔적 교정 | 선택 |

### context

기획서, 요구사항 문서, 코드베이스를 분석하여 도메인 지식을 `context/{도메인}/`에 등록합니다. 문서 기반 생성, Q&A 기반 생성, 코드 스캔, git 히스토리 동기화 중 상황에 맞는 모드를 자동 선택합니다. 등록된 context는 `dev` 실행 시 자동 참조됩니다.

### lens

코드에서 비즈니스 정책을 찾아 PO/PD가 읽을 수 있는 보고서로 출력합니다. 코드를 수정하지 않습니다. 변경 아이디어를 이어서 입력하면 복잡도와 리스크 분석까지 수행합니다.

### dev

자연어 요청 하나로 PRD 작성부터 PR 생성까지 전체 사이클을 수행합니다. "긴급 수정" → hotfix 모드, "PRD만" → 특정 단계만, "이어서 해줘" → 중단 지점부터 재개 등 의도를 자동 판단합니다.

**에이전트 팀:**

| 에이전트 | 역할 | 단계 |
|---------|------|------|
| 제품책임자(PO) | 요구사항 구체화, PRD 작성, 인수 검증 | requirements, complete |
| 설계자 | 기술 설계 (변경 범위, API, 구현 순서) | design |
| 설계 비판자 | 암묵적 가정 도전, 과잉 설계 식별 | design (중형 이상) |
| 개발자 | 설계 기반 코드 구현 | implement |
| QA 매니저 | 코드 리뷰 + 스펙 충족 검증 | implement, review |
| 보안 감사자 | 정책/보안/허점 교차 검증 | review |

## 보조 기능

```
"커밋해줘"          → 브랜치명에서 타입 파싱, 변경사항 분석, 한국어 커밋 메시지 생성
"PR 만들어줘"       → 커밋 히스토리 분석, PR 제목/본문 자동 생성
```

## 안전장치

- PR 생성까지만 자동화합니다. **PR 머지는 사용자가 직접** 수행합니다.
- `git push --force`, `gh pr merge`는 설정 수준에서 차단됩니다.
- 보호 브랜치(main)에서 직접 커밋을 차단합니다.
- 커밋 전 민감 파일(`.env`, `*.key`, `*.pem`, `credentials*`, `*secret*`) 감지 시 경고합니다.
- 빌드 아티팩트(`build/`, `node_modules/` 등)가 tracked 상태이면 자동으로 `.gitignore` 보강을 제안합니다.

## FAQ

<details>
<summary><b>context 없이도 dev를 실행할 수 있나요?</b></summary>

네. 없어도 동작합니다. 다만 context를 등록하면 AI가 도메인 용어를 정확히 이해하여 더 정확한 코드를 생성합니다.
</details>

<details>
<summary><b>dev를 실행하면 바로 코드를 짜나요?</b></summary>

아닙니다. PO 에이전트가 먼저 Q&A로 요구사항을 구체화하고(PRD), 설계자가 기술 설계를 거친 뒤 사용자가 승인해야 구현에 들어갑니다. 각 단계마다 선택형/자유입력형 질문으로 확인을 받습니다.
</details>

<details>
<summary><b>dev 도중에 멈추면 처음부터 다시 해야 하나요?</b></summary>

아닙니다. "이어서 해줘"라고 말하면 `.dev/state.md`에 저장된 진행 상태를 기반으로 중단된 단계부터 재개합니다.
</details>

<details>
<summary><b>PR이 자동으로 머지되나요?</b></summary>

아닙니다. PR 생성까지만 자동화합니다. `gh pr merge`는 설정 수준에서 차단되어 있습니다.
</details>

<details>
<summary><b>플러그인 업데이트는?</b></summary>

`/plugin marketplace update oh-my-gx`
</details>
